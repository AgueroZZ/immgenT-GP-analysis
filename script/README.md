# script-refactor/

One R script per published figure. Each script:

- reads only from `data/` (never writes there),
- writes its panels into `figure-refactor/Figure N/` (or `Figure SN/`) using
  the exact panel filenames used in `figures/final-selected/bits/`, so each
  output can be dropped next to and diffed against the ground truth,
- sources shared plotting/data-loading helpers from `code-refactor/R/`
  rather than redefining them.

Run any script from the repo root, e.g. `Rscript script-refactor/Figure4.R`.

## Verification status (as of this refactor)

Every script above was run end-to-end and its output PDFs compared
byte-for-byte against `figures/final-selected/bits/`. Most panels are
byte-identical; a handful differ by well under 1% (floating-point/label-repel
layout nondeterminism in `ggrepel`, not a logic difference). Panels that
required *reconstruction* (no exact source code existed for them) are called
out below with their approximation quality:

- **Figure 1, panel 1D** and **Figure 2, panels 2D/2F/2H** ("signature
  volcano" plots for GP1/GP68/GP30/GP58): these are **not from `script/` or
  `code/` at all** — they're generated by a separate Shiny app,
  `immgen-signature` (a sibling project, `.../Immgen/webapps/immgen-signature/`,
  outside this repository). The plotting function was ported into
  `code-refactor/R/volcano_helpers.R` (`plot_gp_signature_volcano()`, matching
  `mod_signature.R::.make_static_plot()` exactly). The app lets a user
  interactively set the number of labeled genes (`n_label`) per plot before
  exporting, so the exact value isn't recoverable from code — it was
  recovered empirically here by bisecting against each panel's byte size:
  GP68 (2D) and GP58 (2H) match **exactly**; GP1 (1D) and GP30 (2F) are
  within ~30 bytes (0.01-0.2%).
- **Figure 2, panel 2M** (top-30-gene cross-GP heatmap): from the
  `immgen-signature` app (`mod_cross_gp.R`'s heatmap builder, ported to
  `code-refactor/R/cross_gp_helpers.R`'s `plot_cross_gp_heatmap()`), not a
  reconstruction. The app-faithful version (columns GP3/GP22/GP29, rows
  purely hclust-ordered) matched the published PDF almost exactly (1 byte
  off). The script now uses a deliberately customized presentation instead
  (column order GP3/GP29/GP22, and Fcer1g/Ccl5/Cd7 manually pinned to the
  top 3 rows via the `pin_top` argument -- our own addition, not part of
  the app), so **2M is no longer a byte-exact reproduction of the original
  published PDF** by design.
- **Figure 2, panels 2C/2E/2G** (GP68/30/58 loading on the *global* MDE):
  no exact source existed in `script/Figure_Lineage.R`; reconstructed using
  the same `plot_loadings_on_umap()` styling used for panels 2J/2K/2L.
  Sizes are within ~6-12% of the final PDFs — same content (confirmed
  visually), not byte-identical.
- **Figure S3, panel s3c** (GP26+ rate by organ): reconstructed following
  the same per-organ threshold-rate pattern used in Figure4.R's panel 4b.
  No caption panel-lettering ambiguity here (this is genuinely new code),
  size is within ~10% of the final PDF.
- **Figure 6, panel 6a** (schematic): like Figure 1's panel 1B, this is a
  hand-drawn schematic (confirmed via PDF `/Creator` metadata: `6a.pdf` and
  `1B.pdf` are the only two non-R-generated panels among the 9 figures) --
  not code-generated, no source to port.
- **Figure 6, panels 6c-6f** (protein-gate vs. GP-loading galleries): use
  the genuine `plot_gated_gp_vs_protein()` logic from the original script;
  sizes differ from the final PDFs by 5-30%, most likely from
  `scattermore` raster/density-color nondeterminism rather than a logic
  difference (6b, 6g-6k, which don't rasterize a density plot, are
  byte-identical or within a few bytes).

## Note on re-running FigureS6.R

`cairo_pdf()` was found during verification to not reliably truncate an
existing output file of a different size in place (a stale/partial file
from an earlier interrupted run could persist and get silently reused).
`FigureS6.R` now explicitly `unlink()`s each page's output file before
opening the `cairo_pdf()` device, so re-running it is safe. If you ever see
a script's output that doesn't match expectations, deleting its output
files before re-running is good practice in general.

## Why Figure 7 and Figure S7 are excluded

Confirmed via PDF `/Creator` metadata during this refactor:

- Figure 7, panel A is a **Figma** schematic (not code-generated).
- Figure 7, panels C/D/F and **all 11** Figure S7 panels are **Matplotlib**
  (Python) output — but no `.py` file exists anywhere in this repository.
- The remaining Figure 7 R-generated panels (7B, 7E, 7G) trace to
  `code/replicate_RQVI_cells.R` and related RQVI validation code, but that
  entire validation track was scoped out of this refactor (see
  `code-refactor/README.md`).

These are not reproducible from this repository as it stands. If the
missing Python source turns up, it can be added as a `code-refactor/python/`
sibling; until then, Figure 7/S7 have no script here.
